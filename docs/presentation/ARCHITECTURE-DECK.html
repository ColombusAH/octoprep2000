Problem & Solution

Problem

  • Not enough lecture advisors at Tikal to go around
  • Presenters often skip advising sessions — easy to avoid
  • So many show up to present unprepared
  • No easy way to practice and get feedback alone

Solution

  • Always-on AI co-pilot watches + listens in real time
  • Upload → Record → Analyse → Report pipeline
  • 4-vector scorecard: voice · body · slides · content
  • Timestamp + slide anchored insights · score /100 · mentor unlock ≥80

Agents · contract

5 Agents + 1 Utility

AgentPhaseModel / ServiceEmits
Frame ServiceP1imagehash dhash (algorithmic)filtered frame stream
Audio AgentP1ElevenLabs Scribe v1writes transcript_entries + audio_warnings
Vision AgentP2GPT-4o Vision (LiteLLM)writes video_events (batched 1s/N=20)
PPTX AgentP1LiteLLM · 12-factor Playbookwrites slide_analyses + slides_raw_text
Content AgentP2LiteLLM · reads transcriptwrites content findings
Report AgentP1/P2LiteLLM · reads agreed tableswrites reports (4 vectors)

Golden rule: one writer per table. Each agent owns its role-scoped rows, writes them to Postgres, and only after the commit lands sends a notify_complete completion signal to the Orchestrator. The Orchestrator owns lifecycle + cross-agent coordination and reads the agreed tables to assemble the final report.

Window-based orchestration: the LiveWindowAggregator (one per session) buffers deduped frames; each ~2s audio chunk closes a window and fires one LiveSessionWorkflow run as a detached task. That run is a single agno Parallel("LiveAnalysisTeam") — vision ‖ audio — deliberately a Parallel, not a coordinator Team (the two modalities are fixed and audio is fully deterministic, so a routing leader would only add latency). The workflow is pure in-process glue (telemetry=False, db=None, no DB writes); its steps delegate into the long-lived stateful agents, so per-session state survives across windows.

Two vision paths: instant Google Cloud Vision face detection stays fire-and-forget on frame ingest (sub-100ms, OUTSIDE the window) for fast eye-contact/face signals; the heavier GPT-4o Vision 3-frame posture/gesture batch runs inside the window's vision step. Audio-only fallback is enforced there too — the step checks is_fallback and skips. See workflows/live_session.py + agents/vision_agent.py.

    class D red

  • 60fps would be ruinous on latency + token cost — banned
  • imagehash dhash → Hamming distance vs last frame
  • Distance ≤ 8 → frame is redundant → drop silently
  • Survivors batched 3 per call to amortise tokens (~1000 calls / 10-min session)
  • Result: ≤5fps reaches the Vision Agent
  • Independent second cap: GCV face_detection self-throttles to ~1fps inside VisionAgent (NFR-004), a separate cost layer from this filter

Rationale

Why These Choices

Single FastAPI process

  • All agents = asyncio tasks, zero inter-service networking
  • Right for a 6-hour build · risk: blocking call stalls loop → run_in_executor

In-process pub/sub

  • asyncio.Queue + dict[session_id → set[WS]]
  • No Redis · clean interface, swappable post-hackathon

One writer per table + completion signals

  • Each agent writes its own role-scoped rows, then sends notify_complete (durability before signal)
  • Orchestrator coordinates + reads the agreed tables to build the report · VideoEvents batched 1s/N=20

Capability tokens

  • Full session isolation, no login flow
  • Agno PoC not adopted for the Orchestrator — raw asyncio.create_task instead

Provider fallback (Claude + direct ElevenLabs)

  • Shared LiteLLM gateway is one team token budget, a single chokepoint risk
  • PROVIDER_MODE=auto retries Vision/PPTX/Content on personal-key Claude + STT on direct ElevenLabs after a gateway failure; direct flips the order pre-emptively if the team knows the gateway is exhausted
